# Re-Sync Process: mercure → base-skills

## Replica Relationship

**mercure** (in `ccsetup`) is the canonical SSOT: a full-featured Claude Code plugin with
72+ skills, 11 agents, 5 MCP servers, hooks, and CC-specific primitives (TaskCreate, MCP
resources, worktree isolation, etc.).

**base-skills** is a portable replica: client-neutral skill files with all CC-specific
primitives abstracted away, ready for any agent that supports system-prompt injection or
the agentskills.io spec. The replica inherits behavioral and knowledge content from mercure
but never carries implementation details that only work inside Claude Code.

When mercure skills are updated, this process propagates changes to base-skills.

---

## Three Sync Types

### (a) New Knowledge Skill Port

**Effort**: S (~30 min)
**When to use**: A new `mercure/skills/<domain>-<topic>/SKILL.md` exists that has no
counterpart in `base-skills/skills/`.

**Checklist**:

1. Copy `mercure/skills/<domain>-<topic>/SKILL.md` to
   `base-skills/skills/<domain>-<topic>/SKILL.md`.
2. **H8 cross-reference cleanup** — remove or abstract all mercure-specific references:
   - Strip V-codes (e.g., `V-SOLID-01`, `V-TEST-01`); replace with the `review-gate`
     severity model (`CRITICAL → BLOCK`, `HIGH → BLOCK`, `MEDIUM → WARN`).
   - Remove references to CC-only tools: `TaskCreate`, `TaskUpdate`, `Agent`, `MCP`,
     `WorktreeCreate`, slash commands (`/x-implement`, etc.).
   - Remove `allowed-tools:`, `disallowedTools:`, and `isolation:` frontmatter keys.
   - Keep frontmatter keys: `name`, `description`, `triggers`, `on-demand: true`.
3. Verify the skill self-describes its purpose without requiring knowledge of ccsetup.
4. Add a row for the new skill in `README.md` under the appropriate L2 domain section.
5. Regenerate `CHECKSUMS.md` (see step 8 below).
6. Run `make validate-manual` to confirm no spec violations.
7. Bump the patch version in `README.md` header and `system-prompt.md` comment block.
8. **Regenerate CHECKSUMS.md**:
   ```bash
   cd base-skills
   find skills -name 'SKILL.md' | sort | xargs sha256sum > /tmp/new-checksums.txt
   sha256sum system-prompt.md >> /tmp/new-checksums.txt
   # Verify the new entries look correct, then update CHECKSUMS.md
   ```
   See `CHECKSUMS.md` for the full regeneration one-liner.
9. Commit: `feat(skills): port <name> from mercure`.

---

### (b) Behavioral Skill Update

**Effort**: XS (~15 min)
**When to use**: An existing L1 behavioral skill in mercure (one of the 17 in `rules/`)
has received a content change that should be reflected in base-skills.

**Checklist**:

1. Read the mercure diff for the affected skill file.
2. Apply the same content change to `base-skills/skills/<name>/SKILL.md`,
   removing any CC-specific primitives per H8 cleanup (sync type a, step 2).
3. Check whether the change affects the **Always** or **Never** bullet list in the skill.
   - If yes: update the corresponding block in `system-prompt.md`. The behavioral summary
     block is hand-maintained (not generated for content changes; only the structure is
     generated by `make generate`). Run `make check-drift` after updating to confirm
     the file is consistent.
   - If no: skip `system-prompt.md`.
4. Regenerate `CHECKSUMS.md` (sync type a, step 8).
5. Run `make validate-manual` to confirm no spec violations.
6. Bump the patch version.
7. Commit: `fix(skills): update <name> from mercure <version>`.

---

### (c) Workflow Skill Update

**Effort**: M (~2 hours)
**When to use**: An L3 workflow skill (`x-auto`, `x-analyze`, `x-design`, `x-plan`,
`x-implement`, `x-review`, `x-fix`, `x-troubleshoot`, `x-brainstorm`, `x-research`)
has been changed in mercure.

**Checklist**:

1. Read the mercure diff and the ADR-083 abstraction applicability note for that skill
   (ADR-083 documents which CC primitives were abstracted for each workflow skill during
   the M3 port; see `documentation/decisions/ADR-083-*` in base-skills if present).
2. Assess whether the mercure change:
   - (A) Adds new behavioral content (new phase, new gate, new checklist item) — portable
   - (B) Modifies CC-specific tooling (TaskCreate, worktree, MCP call) — requires re-adaptation
   - (C) Removes content — apply removal directly
3. For (A) and (C): apply the change directly; re-adapt any CC primitives to prose chaining
   per the existing abstraction pattern in the skill file.
4. For (B): identify the closest client-neutral equivalent and update the prose description.
   Do not introduce new CC-only tools.
5. **F0 quality bar check**: The updated skill must demonstrably change agent behavior on
   any agent, not just CC. If the update only makes sense inside CC, defer it and document
   the gap in `references/resync-process.md` (this file) under a new "Deferred Adaptations"
   section.
6. Regenerate `CHECKSUMS.md` (sync type a, step 8).
7. Run `make validate-manual`.
8. Bump the minor version (workflow changes are higher-impact than patch-level).
9. Commit: `feat(workflow): update <name> from mercure <version>`.

## CHECKSUMS.md Regeneration (All Sync Types)

After any file change, regenerate the checksums before committing:

```bash
cd /path/to/base-skills
find skills -name 'SKILL.md' | sort | xargs sha256sum > /tmp/checksums-body.txt
sha256sum system-prompt.md >> /tmp/checksums-body.txt
```

Then update the `## Files` section of `CHECKSUMS.md` with the new hashes and update the
`Generated:` date in the header. The format is one `{sha256}  {path}` line per file
(two spaces between hash and path, matching `sha256sum` output exactly) so that
`sha256sum --check CHECKSUMS.md` passes from the repo root.

Verify before committing:

```bash
sha256sum --check CHECKSUMS.md   # GNU coreutils (Linux)
shasum -a 256 -c CHECKSUMS.md    # macOS
```

Exit code must be 0. Any non-zero exit indicates a file mismatch — investigate before tagging.

## Version Bump Policy

| Change type | Version component | Example |
|-------------|------------------|---------|
| New knowledge skill ported | patch | `v0.2.0` → `v0.2.1` |
| Behavioral skill content update | patch | `v0.2.1` → `v0.2.2` |
| Workflow skill update | minor | `v0.2.2` → `v0.3.0` |
| Breaking change (skill removed or renamed) | major | `v0.3.0` → `v1.0.0` |

Update the version in:
1. `README.md` — the `@vX.Y.Z` pinned install syntax in the Install section
2. `CHECKSUMS.md` — the `Generated:` date (and optionally a version comment)
3. `references/bundles.md` — the `@vX.Y.Z` in every install command
4. Git tag: `git tag -a vX.Y.Z -m "vX.Y.Z — <one-line summary>"`

Push the tag only after human review: `git push origin vX.Y.Z`.

## H8 Cross-Reference Cleanup Cheat Sheet

When porting any mercure skill, scan for these patterns and remove/replace:

| Pattern to remove | Replacement |
|-------------------|-------------|
| `V-SOLID-01`, `V-TEST-03`, etc. | Name the principle: "CRITICAL: violates SRP" or omit if redundant |
| `TaskCreate`, `TaskUpdate`, `TaskList` | Remove or replace with "record progress in a checklist" |
| `Agent(...)`, agent delegation calls | Remove; describe the delegation as "spawn a specialized subagent" |
| `MCP`, `mcp__*` tool references | Remove entirely |
| `WorktreeCreate`, `EnterWorktree` | Remove; describe as "isolated branch workspace" |
| `/x-implement`, `/x-review` (slash cmds) | Replace with skill name without slash: `x-implement`, `x-review` |
| `allowed-tools:` frontmatter | Remove entire key |
| `disallowedTools:` frontmatter | Remove entire key |
| `isolation: worktree` | Remove |
| `mercure-plugin/` path references | Remove or replace with `skills/` |
| `ccsetup/` path references | Remove or omit |

After cleanup, read the skill aloud mentally: it should make sense to a developer using
Cursor, Copilot, or any other agent — not just Claude Code.
